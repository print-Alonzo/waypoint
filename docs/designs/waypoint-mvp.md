## Vision

### 10x Check
The 10x version is a live multi-city optimizer: user pastes a list of places from any city,
the system retrieves live POI data (hours, coordinates), computes optimal order with real
transit data (OSRM or Google Maps API), flags conflicts in real-time, and lets the user
export to Google Calendar or WhatsApp itinerary. The algorithm still shows its work — no
black-box outputs. Target segment: working professionals booking short-break trips 3-4x/year.

### Eureka: Competitive Positioning
The AI travel planner market (NXVoy, iMean, MindTrip, Stippl) has converged on
"AI generates your whole trip" — a trust-destroying approach. Data: 7% of users trust
AI travel tools to be accurate every time (2026). Waypoint's inversion — user provides
the POI list, algorithm optimizes ORDER only, all flags visible — is the precise gap
Wanderlog charges a premium for but does not fully solve (no flags, no transparency).
This is a structural trust advantage, not a positioning choice.

## Accepted Scope (full MVP build list)

### Data Schemas

**`data/pois.json` — per POI record:**
```json
{
  "id": "string (kebab-case slug, e.g. 'fort-santiago')",
  "name": "string (display name)",
  "category": "string (heritage | museum | park | market | church)",
  "lat": "float (WGS84)",
  "lng": "float (WGS84)",
  "open_time": "HH:MM (24h, e.g. '08:00')",
  "close_time": "HH:MM (24h, last entry time)",
  "closed_days": ["string[] (e.g. ['Monday'])"],
  "recommended_duration_minutes": "int (conservative estimate, e.g. 90)",
  "notes": "string | null (optional trust signal)"
}
```

**`data/transit-matrix.json` — combined POI-to-POI and landmark-to-POI:**
```json
{
  "POI_ID_or_landmark_slug": {
    "POI_ID": { "walk": N, "jeepney": N, "grab": N }
  }
}
```
All transit times are one-directional: origin → destination in minutes.
Landmark rows (origin-only, landmark → each POI). POI → landmark NOT needed.
Outlier corrections: annotate with `"_note": "manually corrected from 12 to 25"` as
a sibling field on the affected entry. No separate corrections file.

### Core (from design doc)
- POI selector component: checkbox list of 25-30 Metro Manila POIs with category grouping
- Input form: start time (HH:MM), transport mode (walk/jeepney/Grab), start location (dropdown),
  day of week (Mon–Sun dropdown; required for closed_days red-flag evaluation).
  Validation: submit button disabled when 0 POIs are checked; inline error: "Select at least one place."
- `/lib/scheduler.ts`: nearest-neighbor greedy ordering with time-window feasibility check.
  Algorithm seeds from selected start-location landmark.
  Step 0: compute transit time from start-location to all selected POIs (from matrix);
  pick the minimum as Stop 1. Then apply nearest-neighbor from Stop 1 onward.
  "Nearest" = lowest transit time (minutes) for selected transport mode.
  Ties (transit time difference < 5 min) broken by earlier close_time. Algorithm NEVER drops
  a POI — all POIs appear in output, flagged if infeasible. Flags are advisory only.
- Time computations: all time values within the scheduler are integers (minutes since midnight).
  Parse on load: `"09:00" → 540`. Compute in integers. Format to HH:MM for display.
  `arrival_time = previous_departure_time + transit_time_minutes`
  `departure_time = arrival_time + poi.recommended_duration_minutes`
  For Stop 1: `arrival_time = user_start_time_minutes + transit(start_location, stop1)`
  Yellow flag: fires when `arrival_time > poi.close_time_minutes` (integer comparison)
  Red flag: fires when user-selected day-of-week string is in `poi.closed_days[]`
  After selecting Stop N, remove it from the unvisited set. Repeat until unvisited set is empty.
  Missing matrix entry fallback: if a pair is absent, compute Haversine distance × walk-mode
  multiplier and emit a `console.warn`. Do not throw or crash.
  Duration clamp: use `Math.max(poi.recommended_duration_minutes, 1)` when computing
  departure_time. Prevents a data-entry 0 from collapsing all downstream times.
- `/lib/params.ts`: URL parameter encoder/decoder replacing sessionStorage as the selector→result handoff (eng review D8).
  Exports `ScheduleParams` TypeScript type `{ poi_ids: string[], start_time: string, transport_mode: string, start_location: string, day_of_week: string }`.
  Exports `encodeParams(params: ScheduleParams): URLSearchParams` — serializes all 5 fields for router.push.
  Exports `decodeParams(params: URLSearchParams): ScheduleParams | null` — returns null if any required field is missing or invalid (poi_ids empty, start_time not HH:MM format). Never throws. Caller treats null as invalid URL.
  All pages import ScheduleParams from here — one source of truth, no magic strings.
- `/lib/scheduler.test.ts`: unit tests for algorithm (Vitest). Happy path: 4 POIs, no conflicts.
  Shadow paths: (a) all POIs have selected day in closed_days (all red-flagged, none dropped);
  result view renders all rows red + top-of-page banner: "All selected places are closed on this day."
  (b) start time leaves insufficient time for any POI (all yellow-flagged, schedule shown as-is)
  (c) single POI selected (trivial output: one row, no transit steps, algorithm exits after Step 0)
  (d) duration = 0 (clamped to 1; downstream times differ from arrival_time by ≥1 min)
  (e) tie-break: two POIs with transit delta < 5 min → earlier close_time wins; boundary: delta == 5 min exactly → nearest wins (eng review D5)
  (f) Haversine fallback: POI pair absent from matrix → result computed (no throw), console.warn called with missing pair IDs (eng review D7)
  (g) mixed flag state: some red + some yellow + some clear rows → each row has correct flag styling, no all-red banner
  (h) yellow boundary: arrival_time == close_time → no yellow flag fires (strictly > required)
- `/lib/params.test.ts`: unit tests for URL params module (Vitest) — replaces storage.test.ts.
  (1) encodeParams → decodeParams round-trip: same ScheduleParams object returned
  (2) decodeParams with empty URLSearchParams → returns null
  (3) decodeParams with missing 'poi_ids' field → returns null
  (4) decodeParams with missing 'start_time' field → returns null
  (5) decodeParams with empty poi_ids array → returns null
- Result view: ordered list with per-stop arrival time, visit duration, transit time, status flags.
  When all POIs carry a red flag: render schedule normally with all rows red; display a
  top-of-page banner: "All selected places are closed on this day."
- Flag types: yellow warning (arrival_time > close_time), red error (day_of_week in closed_days)
  Flag styling (accessibility): use row background color, not text/icon color alone.
  Yellow flag row: `bg-yellow-100` background + dark text + ⚠ label "Check hours".
  Red flag row: `bg-red-100` background + dark text + ✕ label "Closed".
  Background shift is the primary signal — readable regardless of color perception.
  WCAG AA compliant (background contrast ratio sufficient for both flag states).
- Print stylesheet (`window.print()`, no JS library). Print triggered by a `<button>`
  labeled "Print itinerary" on the result page; the button calls `window.print()` directly
  (no print library). Print view hides: nav bar, 'Edit this list' button, confidence labels.
  Flag row backgrounds (red/yellow) preserved via `print-color-adjust: exact` in print CSS.
  Safari requires `-webkit-print-color-adjust: exact` — add both properties for cross-browser.
  Page orientation: portrait. Tested in Chrome + iOS Safari before beta.
- Static data: `data/metro-manila/pois.json` + `data/metro-manila/transit-matrix.json`
  City slug read from `NEXT_PUBLIC_CITY=metro-manila` env var (set on Vercel dashboard).
  All scheduler and data imports reference `process.env.NEXT_PUBLIC_CITY` for the subfolder.
  V2 multi-city = add `data/cebu/` folder + update env var on Vercel. Zero code changes.
- Vercel deploy (free tier, direct URL for testers)

- Result page guard: on mount, call `decodeParams(searchParams)` from `/lib/params.ts`.
  If null (missing or invalid URL params): call `router.replace('/')` immediately.
  If valid: compute itinerary from decoded params and render.
  Handles direct URL access, bookmark, page refresh (URL still valid → re-computes identically).
  Mobile tab-kill behavior: URL is preserved by the browser — user reopens /result, URL still has params, itinerary re-computes. No guard fires for a valid URL. Tester brief should note this is reliable, unlike sessionStorage.
- React Error Boundary wrapping the result page component. On any uncaught render or computation
  error: show "Something went wrong. Go back and try again." with a link to '/'. Prevents white
  screen during evidence collection sessions. ~20-line component in `components/ErrorBoundary.tsx`.

## Dependency Note

D3.5 (start-location dropdown) requires extending the transit matrix with origin rows for each
landmark → all POIs. ~375 values (5 landmarks × 25-30 POIs × 3 modes).
Matrix is one-directional: landmark rows are origin-only (landmark → POI).
Format: `{ "Rizal Park": { "POI_ID": { "walk": N, "jeepney": N, "grab": N } } }`.
POI → landmark rows are NOT needed (the route ends at the last POI, not a landmark).

Computation method: Node.js script using Haversine distance × transport speed multiplier.
Data source: computed from POI WGS84 coordinates + assumed speed multipliers. NOT scraped
from Google Maps API or any live routing service (no ToS or cost risk). Haversine is the
initial estimate; OwnRide/Google Maps are used only for the 20% spot-check validation step.
Assumed multipliers (Metro Manila urban conditions):
  walk: 4 km/h effective (sidewalks, pedestrian crossings)
  jeepney: 12 km/h effective (frequent stops, heavy traffic)
  Grab: 20 km/h effective (direct route, typical Manila traffic)
Script output: draft `transit-matrix.json`. Multipliers calibrated against 20% spot-checks
in Weeks 4-5 against OwnRide/Google Maps. Accept ±15% error margin.
Corrections annotated inline via `_note` field in transit-matrix.json. No separate corrections file.
Computation timeline: run the matrix script in Week 2 (parallel with POI selector build).
Spot-check 20% of values in Weeks 4-5 (parallel with internal alpha). Week 6 = corrections
only, not initial generation. This prevents the data task from colliding with beta launch.

Assign to: same team member handling POI curation. Budget ~3 hours for spot-checks (75 lookups × ~2 min each). Allocate dedicated time in Week 4 sprint — do not let it spill into Week 5 alpha testing.
Hard gate: matrix must be complete and spot-checked before Week 6 beta launch.
Blocker status for dev: NOT a blocker. Build UI with placeholder coordinates; script populates separately.

## Reviewer Concerns (unresolved after 3 adversarial review iterations)

After 3 rounds of adversarial spec review, the following are residual low-risk judgment calls
rather than implementation blockers. They are noted for the team to be aware of, not action items.

1. **Survey question validity (open judgment):** The post-session survey question "Did you save
   or share this itinerary?" is a stronger behavioral proxy than "Did you use it on a trip?"
   but is still stated preference. True behavioral evidence (e.g., screenshot provided, itinerary
   followed on a real trip) requires follow-up with testers post-trip. Plan for a 2-week follow-up
   message from Evidence Lead to Week 7 testers who confirmed intent to travel.

2. **sessionStorage multi-tab limitation — RESOLVED by eng review:** URL params replaced sessionStorage as the result-page handoff (D8). Each browser tab has its own URL; no clobber. The /result URL is shareable and bookmarkable. This concern is closed.

3. **Transit multiplier calibration:** The assumed speed multipliers (walk 4 km/h, jeepney
   12 km/h, Grab 20 km/h) are reasonable estimates for Metro Manila but will diverge from
   real transit times in peak hours. ±15% acceptance threshold mitigates this but does not
   eliminate it. Testers should be briefed that transit times are estimates.
